## 一、全局设计规范

### 1.1 提示格式规范(取代当前 6 种混用变体)

**现状变体盘点:**

| 变体 | 出现处 | 例子 |
|---|---|---|
| A `[回车X,k=Y]` 方括号+半角逗号 | confirm | `[回车不显示,e=显示]` |
| B `[输入 1-N,0 返回]` 方括号+逗号 | pick | `[输入 1-5,0 返回]` |
| C `[default] · 回车X · 0 返回` 中点 | line | `[Yellow Coldplay] · 回车搜索 · 0 返回` |
| D 文中括号 `(回车跳过)` / `(可选,…;0=返回…)` | audioFlow line | `粘贴歌曲 URL,或输入歌名搜索(回车跳过)` |
| E 裸文本 `回车=下一页,s=跳到保存,0=返回候选` | 预览翻页 | 无括号无中点 |
| F 全局图例 `全局: 回车执行默认动作 · 0 返回(可用时) · q 退出 · Ctrl+C 中断` | GLOBAL_PROMPT_HELP | — |

**统一规范(一套,三类问答共用):**

```
<问题文本> [<默认值,仅数据类>] · <键 动作> · <键 动作>:
```

规则:
- **分隔符**:段与段一律 ` · `(空格-中点-空格)。彻底弃用图例里的半角逗号和 `=`。
- **快捷键标注**:统一 `<键> <动作>` 两段式(空格分隔),不再用 `键=动作`。键写作 `回车 / 0 / r / d / s / q`,动作是动词短语(`保留`/`重新下载`/`返回`)。
- **方括号 `[]` 只用于展示"默认数据值"**(路径、关键词),永不用于键图例。
- **圆括号 `()` 只用于问题文本内的说明**(如"歌手(可选)"),不用于键提示。
- **顿号 `、`** 仅用于问题正文的中文并列;提示段不用。
- 冒号 `: ` 收尾,前面无空格。
- 提示文本尽量陈述句式,问句仅在确认题保留 `?`。

### 1.2 颜色与符号规范(term.mjs)

**现状:** `emit` 打 `● `(按 kind 上色:info 蓝39/start 橙/success 绿/warn 黄/error 红),`detail` 打 `└ `(dim)。交互提示(readline question)完全无色——这正是"命令提示 vs 命令结果"无法区分的根因。

**统一三符号体系:**

| 符号 | 语义 | 颜色 | 来源 |
|---|---|---|---|
| `●` | 状态/结果输出 | 按 kind | 已有 emit |
| `└` | 结果的子项明细 | dim | 已有 detail |
| `?` | **需要用户输入的提问** | cyan(36) | **新增** |

- 提问行结构:`<cyan>?</> 问题文本 <dim>· 键 动作 · 键 动作</dim>: `——问题文本用常规色,**键图例整体走 dim**,让视觉焦点落在问题上。
- term.mjs 新增能力:导出 `paint(kind, text, stream)`(按 ansiEnabled 决定是否包 ANSI)和 `promptPrefix`/`dim(text)`,供 prompts.mjs 复用同一套 ansiEnabled 判断,不重复实现。
- 非 TTY / NO_COLOR / TERM=dumb 时:`?` 前缀保留为纯文本,键图例不加 dim,与现有 emit 降级一致。
- **风险**:ANSI 包裹的 readline prompt 可能影响行编辑光标计算。缓解:只给前缀和图例上色,问题主体+默认值+冒号保持无码;真实 TTY 冒烟验证退格/中文输入正常。

### 1.3 按键规范(减少击键,回车走安全默认)

保留 2026-07-15 persistent-cli-menu 已确立的核心原则:**回车=安全且最常用动作;破坏性/联网重试保持显式键**。在此之上做两类收敛:

- **消除冗余的全局图例**:每个提问已自述可用键,`GLOBAL_PROMPT_HELP` 整行退休,不再在入口和每个子流程重复打印。仅在**菜单首屏**保留一行极简 dim 提示:`回车 选默认 · 0 返回 · q 退出`(去掉"Ctrl+C 中断"——通用常识;去掉"(可用时)"括注)。
- **默认值前置**:凡是能从上下文推出的输入,预填为默认值,让用户回车即采纳(见 §二逐点表)。

### 1.4 启动 Banner 与退出画面(ASCII)

只在**裸命令进入交互菜单**时打印 Banner;直接命令(带参数/管道)绝不打印,保持可脚本性。退出画面只在从菜单 `q` 正常退出时打印。宽度 ≤ 40 列,和风+猫。用 start 橙色(38;2;217;119;87),降级为纯文本。

启动 Banner(推荐极简版,规避全角对齐在窄终端错位的风险):

## 四、点名丑陋提示的重设计

**其一——全局图例:**
```
旧:  全局: 回车执行默认动作 · 0 返回(可用时) · q 退出 · Ctrl+C 中断
```
→ 整行删除。理由:每个提问已自述可用键,该行是纯冗余记忆负担。仅菜单首屏保留(dim):
```
新:  回车 选默认 · 0 返回 · q 退出
```

**其二——路径题:**
```
旧:  输入素材文件夹路径,或拖入后回车 · 回车返回 · 0 返回:
```
问题:"拖入后回车"(有内容=提交)与"回车返回"(空=返回)让"回车"承担两义,再叠一个 `0 返回` 三重冗余。
```
新:  ? 拖入或输入「素材文件夹」路径 · 回车留空返回:
```
只留一种返回(空回车),删掉 `0 返回`。带 `?` cyan 前缀,图例 dim。

### 实施中发现的问题与根因

1. **runFetch 测试偶发拖垮整个 fetch.test 文件**(`Unable to deserialize cloned data`):runFetch 内 term 直写 process.stdout,与 node:test 子进程的报告流异步交错。第一版修复(打桩 `process.stdout.write`)更糟——runner 缓冲的 TAP 结果在打桩窗口内被吞,29 个测试只上报 3 个且静默不报错。根因是打桩了共享通道而非输出源;最终改为打桩 term 单例的 6 个方法,通道不受影响。教训:验证时对比"声明的 test() 数 vs 实际执行数",光看 pass/fail 汇总会漏掉整文件级的静默截断。

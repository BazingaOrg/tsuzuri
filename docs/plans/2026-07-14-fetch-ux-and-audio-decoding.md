# fetch 交互收尾与音频解码修复

> 状态:已实施并验证。

已确认决策(2026-07-14):

1. 退出即清理临时下载,不做跨会话缓存;下次重新下载。
2. fetch 新素材写入 `audio/`;用户自备素材可放根目录或 `audio/`,旧项目不强制迁移。

## 背景与已确认现状

1. yt-dlp 成功后先写入系统临时目录 `tsuzuri-fetch-*`;只有歌曲信息最终确认后才安装到
   素材文件夹。正常完成会在 `finally` 删除临时目录,但当前 SIGINT/EOF 直接
   `process.exit(130)`,可能绕过 `finally`;进程一直停在输入题时,文件也会一直保留。
2. 下载文件的 title tag 实际常是视频标题。当前把它直接作为歌曲名默认值,容易把频道文案、
   MV 标记等写入文件名和歌词搜索条件。
3. 歌手不是生成音频文件的必填项,但能显著改善 LRCLIB 匹配;现有「可留空」没有解释用途,
   且进入歌手题后只能走到最终确认再返回歌曲名。
4. `[Y/n]` / `[y/N]` 用英文大小写暗示默认值,虽是 CLI 惯例,但与中文的
   `回车=是/否`重复且不直观;各问题的默认值也缺少一条用户可预测的规则。
5. LRCLIB 只打印前 12 行歌词,无法在保存前完整检查。
6. 对 M4A/AAC,`librosa.load` 会先触发 PySoundFile 失败,再退回已弃用的 audioread。
   本地复现仍能完成节拍分析,因此用户贴出的两行是警告而非最终错误;但 librosa 1.0 将删除
   audioread 支持,需要修复真实解码路径,不能只屏蔽警告。
7. planner 和渲染计划把「每张照片的平均展示时长」写成「人均」,概念不准确。
8. fetch 下载的音频与 LRC 直接写在素材文件夹根目录,与照片混排;`metadata/`、`output/`
   已分目录,在线获取的歌曲素材也应有稳定归档位置。

## 设计决策(推荐方案)

### 1. 临时下载只活在当前进程,退出即清理

- 不引入跨会话 pending 缓存;没有隐藏状态、过期时间和并发清理规则。
- `withPrompts` 不再直接 `process.exit`;SIGINT/EOF 改为抛出专用取消错误,让调用栈正常展开,
  `audioFlow` 的 `finally` 得以删除下载目录。顶层识别该错误,安静地以 130 退出。
- 进程仍停在歌曲信息问答时,临时文件必须保留,因为确认后还要安装;用户取消或进程正常退出
  才清理。SIGKILL、断电等无法执行进程内清理的情况仍由系统临时目录生命周期兜底。
- 取舍:退出后再次运行需要重新下载。这比在素材夹长期保留未确认音频更可预测。

### 2. 歌曲信息不再把视频标题当默认答案

下载完成后先单独展示来源信息,再收集真正的歌曲信息:

```text
来源视频: 周杰伦 - 晴天 Official MV ...
歌曲名(必填,用于文件名和歌词搜索):
歌手(可选,填写后歌词匹配更准确;0=返回修改歌曲名):
└ 保存文件: 晴天 - 周杰伦.m4a
歌曲信息正确吗? [回车确认,n=返回修改]
```

- 视频标题只作参考,不再进入歌曲名输入框的默认值。
- 歌手题无默认值:回车明确表示不填;`0` 返回歌曲名,不再承担「清空默认歌手」的双重语义。
- 去掉 `将保存为:` 前的两个手写空格,改用统一 detail 行,与其他状态输出对齐。

### 3. 确认题改用中文直白键位,默认值遵循流程阶段

- 删除 `[Y/n]` / `[y/N]` 大小写惯例:
  - 默认是:`[回车确认,n=否]`
  - 默认否:`[y=是,回车跳过]`
- 默认值规则:
  - 用户刚刚检查过的结果(歌曲信息、歌词预览)→ 回车确认。
  - 启动可选步骤、覆盖/删除、失败重试 → 回车跳过/取消。
- 因此 runFetch 的「现在下载」「在线搜索歌词」也改为默认否;显式输入 `y` 才启动联网步骤。
  这与 offerFetch、替换、重试保持一致,用户无需记住入口差异。

### 4. 用用户语言解释本地歌词识别

将「搜不到会用本地 Whisper」统一改为:

> 没找到在线歌词也可以继续;渲染时会在本机识别人声并生成字幕,不会上传音频。

技术名 Whisper 只保留在详情/文档中,不作为主交互文案的前置知识。

### 5. 歌词候选支持完整分页预览

- 每页 12 行,页头显示 `歌词预览 1-12/共 N 行`。
- 未到末页:`回车=下一页,s=保存,0=返回候选`;最后一页进入保存确认。
- 候选选择与预览做成内层循环:返回候选或拒绝保存不会重新请求 LRCLIB。
- 不调用 `less` 等平台命令,保持 macOS/Linux/Windows 一致。

### 6. 音频读取显式走 soundfile + FFmpeg fallback

- 新增 analyzer 内部 `load_audio`:
  1. WAV/FLAC/MP3/OGG 等 soundfile 可读格式直接读取并转单声道;
  2. M4A/AAC 等不支持格式先由 FFmpeg 解码为临时 PCM WAV,再由 soundfile 读取;
  3. 临时 WAV 使用 Python `TemporaryDirectory`,成功或失败都清理。
- `analyze_beats` 不再调用 `librosa.load`,从根源移除 audioread fallback 警告。
- `--lyrics-only` 获取时长也改用同一受控路径或 ffprobe,避免在另一入口重现警告。
- 不简单过滤 warning:librosa 官方已声明 audioread 在 1.0 删除,隐藏警告会留下升级故障。

### 7. 「人均」统一改为「平均每张」

- 用户输出统一为:
  `裁剪模式:歌长图少,在 151.0s 重拍处截断并淡出,平均每张 7.9s`。
- 快闪提示、最终 plan 摘要、CLI 渲染计划和 `docs/config.md` 同步改为
  「平均每张展示时长」;内部变量 `avg` 与配置键保持不变,避免无意义的数据契约迁移。
- 历史计划文档保留原文,只改当前行为和稳定参考文档。

### 8. fetch 下载素材归档到 `audio/`

新项目结构:

```text
osaka-trip/
├── photo-01.jpg
├── photo-02.jpg
├── audio/
│   ├── 晴天 - 周杰伦.m4a
│   └── 晴天 - 周杰伦.lrc
├── metadata/
└── output/
```

- fetch 新下载的音频和在线歌词统一写入 `<folder>/audio/`;目录按需创建。
- scanFolder 同时扫描根目录与 `audio/`,返回相对素材根目录的路径
  (`audio/晴天 - 周杰伦.m4a`),因此旧项目继续工作,不自动搬动用户已有文件。
- 替换根目录旧音频/歌词时,经确认后新文件写入 `audio/`,旧文件才删除,相当于自然迁移;
  取消或失败仍保留旧文件。
- analyzer 写入 beats.json 的 `audio` 字段必须保留 `audio/...` 相对路径,planner 再原样写入
  timeline;Remotion 的 public dir 仍是素材根目录,可直接读取嵌套路径。
- 照片仍只从素材根目录读取;`audio/` 不作为通用递归素材目录,避免无界扫描和同名歧义。
- 根目录与 `audio/` 合计仍必须只有一个音频、最多一个 LRC;冲突时列出相对路径并要求选择/
  清理,不静默决定优先级。

## 风险与边界

- 不做断点续传:取消后重跑会重新下载,这是为换取无持久缓存状态的明确取舍。
- SIGKILL/断电无法保证进程内 finally;系统临时目录可能短期残留,不做可能误删并发任务的
  全局扫描清理。
- FFmpeg fallback 会为不支持格式产生一次临时 PCM 文件,峰值磁盘占用高于压缩音频;
  仅在 soundfile 直读失败时触发。
- `audio/` 是新增约定但不强迁移旧项目;双位置兼容会让扫描逻辑稍复杂,测试需覆盖根目录、
  新目录和两处冲突三种状态。
- 用户贴出的输出只有 warning,没有最终异常或退出码;若其后还有真正的错误,需保留最后
  20 行日志另行定位,不能把所有分析失败都归因于这两个 warning。

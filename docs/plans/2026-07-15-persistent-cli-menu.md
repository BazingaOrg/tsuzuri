# 常驻 CLI 菜单与交互按键收敛

> 状态:已实施并验证。

## 问题与诊断

1. 裸运行 tsuzuri 时，菜单只负责收集一次 argv；具体命令执行完成后进程退出。用户若继续输入
   数字，输入实际已交给 shell。
2. 当前 zsh 配置存在 `3='cd -3'`。因此菜单退出后输入 `3` 会被解释成跳转到目录栈第 3 项，
   目录栈不足时出现 `cd: no such entry in dir stack`。菜单内选择 3 的路由本身已本地复现正常。
3. 当前确认题同时存在 `[回车确认,n=否]` 与 `[y=是,回车跳过]`；默认值安全，但用户需要在
   “是”与“默认动作”两套概念之间切换。
4. 带默认值的输入题只显示 `[默认值]`。例如 `歌词搜索关键词 [Yellow Coldplay]:` 的实际
   行为是回车采用默认词并搜索，但界面没有说明，也没有返回键；`q` 目前只在主菜单有效，
   子流程的全局退出只能依赖 Ctrl+C/EOF。

## 推荐方案

### 1. 裸命令进入常驻菜单

- 仅 `node cli/tsuzuri.mjs` 的 TTY 模式进入循环；带参数的直接命令仍执行一次后退出，保持脚本
  兼容。
- 每个菜单流程成功、失败或用户取消当前子流程后，显示简短结果并返回主菜单。
- 主菜单输入 `q` 才正常退出；直接命令中的 Ctrl+C 仍保持退出码 130。
- 每次重绘菜单前加清晰分隔，不清屏，保留刚才的日志与错误，便于排查。
- doctor、fetch、lyrics、still 和渲染都走原有命令实现，不复制业务逻辑。

### 2. 暂不引入 Vim 式全屏 TUI

- 保留数字选择和 `q`，菜单底部提示 `1-5 选择 · q 退出`。
- 不在本轮加入方向键、Enter 高亮选择或 Esc：这些键需要切换终端 raw mode；路径拖拽、中文
  输入、Ctrl+C 清理和各子流程仍使用 readline，混用会扩大终端兼容与状态恢复风险。
- 若常驻数字菜单实测仍显笨重，再单独评估 TUI 库；不为一排视觉提示先引入依赖。

### 3. 确认题改为“回车执行安全默认动作”

- 不把所有问题机械改成“回车=是”，否则删除、覆盖、重试和联网会默认执行。
- 去掉抽象的 `[y=是,回车跳过]`，直接写具体动作，例如：

```text
没有音频 [回车跳过,d=下载]:
已有 song.m4a [回车保留,r=重新下载]:
歌曲信息正确吗 [回车确认,r=修改]:
歌词正确吗 [回车保存,r=返回候选]:
```

- 回车始终代表安全且最常用的默认动作；其他键只触发提示里明确写出的动作。未知键原地提示，
  不把拼写错误静默当成“否”。
- 删除多个音频等不可逆操作继续要求显式按键，不因统一视觉而降低保护。

### 4. 所有交互题显示统一导航键

- 主菜单和每个子流程入口都显示一行紧凑图例：

```text
全局: 回车执行默认动作 · 0 返回 · q 退出 · Ctrl+C 中断
```

- 这行图例在普通滚动终端中随内容滚动，不模拟固定底栏；每次返回主菜单或进入新子流程时
  重新显示，确保用户不必依赖记忆。真正固定在窗口底部需要全屏重绘和 raw mode，不纳入本轮。
- `回车`：执行当前提示明确写出的默认动作，不再让用户根据括号猜测。
- `0`：在存在合理上一步时返回；没有上一步的题不显示也不接受 `0`。
- `q`：从主菜单和任意子流程退出整个 tsuzuri，退出码为 0；Ctrl+C 仍表示中断，退出码 130。
- 列表、确认和文本输入共用这套导航语义；业务输入若可能合法等于 `q`，只在该题禁用快捷键
  或提供明确的转义方式，避免静默吞掉用户数据。
- 单个问题不机械重复整条全局图例，只补充该题动作，例如 `回车搜索 · 0 返回`；`q` 与
  Ctrl+C 的语义由入口图例统一说明。
- 歌词关键词题改为类似：

```text
歌词搜索关键词 [Yellow Coldplay] · 回车搜索 · 0 返回 · q 退出:
```

- 从关键词题按 `0` 返回歌词候选的上一级流程；在 `fetch` 中即返回音频/歌词准备流程，不发起
  LRCLIB 请求，也不删除已有文件。

## 实施步骤

1. 将一次性 `main` 拆成可重复调用的命令分派函数，但保持直接命令行为不变。
2. 裸命令增加菜单循环和流程级错误边界；当前流程结束后返回菜单，`q` 退出。
3. Prompt 层支持动作型提示、严格键位校验、可选 `0` 返回和全局 `q` 退出；迁移 menu/fetch
   的确认题与歌词关键词题。
4. 增加测试：菜单 3 正确路由、连续执行两个流程、流程失败后仍回菜单、任意子流程 q 退出、
   歌词关键词回车采用默认值、0 返回不联网、直接命令一次性行为、危险操作不因回车执行。
5. 更新 README、项目状态和本计划实施记录。
6. 全量验证：CLI tests、analyzer tests、renderer typecheck、真实 TTY 冒烟测试、diff check。

## 风险与边界

- 渲染流程结束后回菜单会让进程继续驻留，这是裸命令的预期变化；直接命令不受影响。
- 流程级异常必须清理自己的临时资源后再返回菜单，不能在外层吞掉 cleanup。
- 本轮不支持鼠标、全屏重绘、方向键和 Esc，避免 readline/raw mode 状态互相污染。
- 菜单循环不能把非零命令结果伪装成成功；应显示错误后再返回菜单。

- 普通流程的“取消”沿用各子流程现有的安全返回值；Ctrl+C 明确保留为整个程序中断，没有改成
  “取消当前流程并回菜单”。

## 自审记录(2026-07-15)

1. 用户看到的 `cd: no such entry in dir stack` 已确认来自 zsh 的 `3='cd -3'`，根因是旧菜单
   已退出后 shell 接管数字输入；常驻菜单修复了触发条件，lyrics 的菜单路由本身无需改动。
2. 检查全局退出资源安全：`PromptQuitError` 与 SIGINT 一样通过异常展开，音频下载和 staged
   安装的 `finally` 不会被绕过；新增 q 触发 caller cleanup 的测试。
3. 检查危险动作：删除多余音频使用 `d`，重新下载/搜索使用 `r`，下载使用 `d`，在线歌词
   搜索使用 `s`；回车分别取消、保留或跳过。
4. 检查兼容性：非 TTY 裸命令仍走 usage，带参数命令不进入循环，现有命令行和脚本行为不变。
5. 未发现遗留阻断问题；方向键、Esc 和固定底栏继续作为明确边界，不在本轮实现。
